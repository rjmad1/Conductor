# Task Decomposition Standard — Conductor Platform

This standard defines the hierarchical breakdown structure, complexity sizing, risk assessment, and dependency modeling standards for planning and executing engineering tasks.

---

## 1. Required Work Hierarchy

To ensure traceability from business goals to code execution, all tasks must conform to the following 8-tier hierarchy:

```
[ Initiative ] ──► [ Epic ] ──► [ Feature ] ──► [ Story ] ──► [ Task ] ──► [ Subtask ] ──► [ Validation ] ──► [ Evidence ]
```

1.  **Initiative:** Multi-quarter strategic goal (e.g. "Build the Conductor B2B messaging MVP").
2.  **Epic:** High-level system domain or capability (e.g. "Temporal Workflow Runtime integration").
3.  **Feature:** Functional user-facing capability (e.g. "WhatsApp campaign opt-out keywords handling").
4.  **Story:** Specific developer deliverable (e.g. "Develop WhatsApp webhook endpoint to receive STOP keywords").
5.  **Task:** Isolated code change or configuration task (e.g. "Add NATS publisher route in messaging domain").
6.  **Subtask:** A single micro-step (e.g. "Verify Hibernate model contains `@Convert` JPA decorator").
7.  **Validation:** Automated check verifying the correctness of the change (e.g. "Verify unit test suite executes cleanly").
8.  **Evidence:** Traceable proof generated by validation checks (e.g. JUnit test output log files).

---

## 2. Sizing Standards

Tasks must be sized to limit agent scope and prevent drift:
*   **Maximum Scope Size:** No single story or task assigned to an agent may exceed 300 lines of code modifications or 4 hours of target execution time.
*   **Sizing Matrix:**
    *   *Small (S):* Single file tweak, documentation update, simple test case additions.
    *   *Medium (M):* Multi-file edits within a single domain package (e.g. adding a model and repository).
    *   *Large (L):* Out of scope for a single agent iteration. Large stories must be decomposed into multiple Medium tasks before execution begins.

---

## 3. Risk Scoring Matrix

Before a task is assigned to an agent, a risk score must be computed:

$$\text{Risk Score} = \text{Complexity (1-5)} \times \text{Impact (1-5)}$$

| Score Range | Risk Tier | Agent Autonomy | Required Approval Gate |
| :--- | :--- | :--- | :--- |
| **1 - 4** | Tier 1 (Low) | Autonomous | Automated PR check validation. |
| **5 - 9** | Tier 2 (Moderate) | Operator | Review and approval by peer developer. |
| **10 - 16** | Tier 3 (High) | Contributor | Approval by Domain Technical Lead. |
| **17 - 25** | Tier 4 (Critical) | Observer | Explicit EDRB board review and sign-off. |

*   *Complexity:* Measures system density, number of files touched, and security parameters.
*   *Impact:* Measures tenant exposure, data loss probability, or configuration shifts.

---

## 4. Dependency Modeling

Task planning must explicitly model dependencies:
*   **Predecessors:** Dependencies must be listed and verified as complete. An agent must not write code for a feature if its dependent API specifications or schemas are still in draft state.
*   **Locking:** If a task requires modifying a shared package, it must declare lock holds on that directory, preventing parallel agents from triggering conflicts.

---

## 5. Acceptance Criteria (AC)

Every Story and Task must contain verifiable, concrete Acceptance Criteria:
*   **No Placeholders:** AC must not contain words like "improve," "secure," or "optimize" without measurable metrics.
*   **Format:** AC must follow the Gherkin-like structure:
    *   *Given:* Preconditions (e.g. "Given active Keycloak tenant session").
    *   *When:* Action trigger (e.g. "When HTTP POST is sent containing STOP body").
    *   *Then:* Expected outcome (e.g. "Then database writes consent status as OPT_OUT and publishes event").
