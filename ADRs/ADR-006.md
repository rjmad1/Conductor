# ADR-006: Authentication & Authorization

## Status
Approved

## Context
Conductor requires secure, role-based access to dashboard features and developer APIs. Tenant users must be authenticated, and their operations restricted by scope. Additionally, we need a mechanism for external APIs to authenticate developer requests. 

Building custom sign-up, password hashing, password reset, Multi-Factor Authentication (MFA), and session management logic increases security risks, invites compliance audits, and consumes substantial engineering hours.

## Decision
We will **Adopt Keycloak** as the self-hosted Identity and Access Management (IAM) provider, integrated with **Kong API Gateway** and **Spring Security**:
1.  **OAuth2 / OIDC Standards:** Keycloak acts as the OpenID Connect Identity Provider. 
2.  **Tenant Separation via Realms:** Each Conductor tenant is mapped to a dedicated Keycloak Realm, segregating client credentials and user groups.
3.  **Gateway Token Verification:** Kong API Gateway intercepts client requests, performs cryptographic signature verification (RS256) of Keycloak JWT tokens, and extracts token claims.
4.  **Header Propagation:** Kong injects validated claims as HTTP headers (`X-Tenant-ID`, `X-User-ID`, `X-User-Roles`) to downstream backend services. Spring Security filters convert these headers into the local thread SecurityContext.
5.  **Role-Based Access Control (RBAC):** We define the following target roles:
    *   `OWNER`: Complete tenant administration, billing, and system access.
    *   `ADMIN`: Configuration modification, integration setup, user management.
    *   `MANAGER`: Campaign management, template creations, customer uploads.
    *   `AGENT`: Customer registry access, 2-way chat console operations.
6.  **Secure API Keys:** Developer API access will utilize custom API keys.
    *   Key format: `ck_` prefix followed by 32 alphanumeric characters.
    *   Storage: Only the bcrypt hash of the API key is stored in the database.
    *   Validation: Dynamic comparison using the first 8 clear-text characters for fast indexing, followed by constant-time bcrypt verification.

## Alternatives Considered
1.  **SaaS Providers (Auth0 / Okta):** Rejected. Monthly licensing costs scale rapidly with active users, which undermines margins for SMB audiences.
2.  **Custom Auth Schema:** Rejected. Developing custom OIDC/OAuth2 layers, secure session tokens, and MFA from scratch creates high maintenance overhead.

## Consequences
*   **Standard Compliance:** Security is backed by Keycloak's tested OIDC architecture.
*   **Zero Licensing Fees:** Operating cost is restricted to container compute charges.
*   **Tenant Identity Isolation:** Dedicated realms prevent cross-tenant credential issues.

## Risks & Mitigations
*   **Risk:** Keycloak introduces a heavy runtime compute requirement (~512MB RAM minimum).
*   **Mitigation:** Run Keycloak in a dedicated AWS ECS Fargate container, isolated from the monolith's compute boundaries, pointing to the RDS PostgreSQL instance.
