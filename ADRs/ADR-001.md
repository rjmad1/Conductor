# ADR-001: Target Architecture

## Status
Approved

## Context
The Conductor platform requires a system design that balances long-term horizontal scalability (handling millions of automated customer notifications per day) with short-term operational simplicity (deployable and maintainable by a small team of 3–4 engineers). 

The initial specification outlined 9 separate microservices. However, deploying, configuring, and monitoring 9 distinct Java runtimes during the MVP phase introduces severe operational complexity, increases cloud resource costs, and slows down developer velocity.

## Decision
We will adopt a **Modular Monolith** architectural style using **Java 21 / Spring Boot 3.x with Virtual Threads (Project Loom)** for the custom business logic of Conductor. 

The application will be divided into clean, decoupled package structures (e.g. `com.conductor.tenant`, `com.conductor.customer`, `com.conductor.workflow`) mirroring the domain boundaries. 
*   All custom modules run inside a single JVM process and deploy as a single container.
*   Communication between modules will occur asynchronously via **NATS JetStream** event publishing or synchronous Spring Application Event publisher interfaces.
*   Direct cross-module database tables joining is strictly prohibited.
*   Core infrastructure components (Keycloak, Temporal Server, Metabase, NATS JetStream) will be adopted out-of-the-box as containerized services, rather than being built from scratch.

## Alternatives Considered
1.  **Distributed Microservices (9 distinct services):** Rejected for Phase 1. The infrastructure cost of running 9 ECS containers, managing inter-service HTTP clients, distributing trace headers, and coordinating distributed transactions across databases is too high.
2.  **Spaghetti Monolith:** Rejected. Standard single-database monostructure without strict boundary controls leads to tight coupling, making future scalability or extraction of bottleneck services (like WhatsApp adapter processing) impossible.

## Consequences
*   **Operational Simplicity:** A single Docker container handles all custom business flows. CI/CD pipelines require only one build and deploy chain.
*   **Developer Sandbox:** Developers can run the entire custom backend locally by launching a single JAR, alongside lightweight Docker containers for Keycloak, NATS, and Temporal.
*   **Path to Microservices:** By enforcing strict module isolation (no DB joins, communication via events), any module can be extracted into an independent microservice in the future with minimal refactoring.

## Risks & Mitigations
*   **Risk:** Package coupling. Developers may import classes from other modules directly, bypassing the asynchronous boundary.
*   **Mitigation:** Enforce strict architectural validation using **ArchUnit** tests in the Maven/Gradle build phase to automatically fail builds that violate package containment rules.
