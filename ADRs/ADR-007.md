# ADR-007: Integration Framework

## Status
Approved

## Context
Conductor workflows must integrate with external systems. This includes:
1.  **Ingress Webhooks:** Shopify order events, Zoho CRM lead generation events.
2.  **Egress Webhooks:** Dispatching customer-defined HTTP calls or updating Zoho/Shopify records.

Executing arbitrary outbound network requests from a backend application server introduces a major vulnerability: **Server-Side Request Forgery (SSRF)**. Attackers could configure webhooks targeting internal cloud IP ranges (e.g. AWS metadata endpoints like `169.254.169.254` or internal database subnets), exposing sensitive system configurations.

## Decision
We will establish a secure **Egress Forward Proxy (Squid)** configuration for the Integration Framework:
1.  **Mandatory Proxy Routing:** All outbound HTTP requests initiated by the integrations engine (including Zoho, Shopify, and custom tenant webhooks) are routed through a dedicated forward proxy cluster running **Squid**.
2.  **Strict Egress Allowlist:** The Squid proxy will reject requests targeting private, local, or loopback IP ranges (e.g., `10.0.0.0/8`, `192.168.0.0/16`, `127.0.0.1/8`, `169.254.169.254/32`). Only outbound calls targeting resolved public internet domains or domains explicitly added to the tenant integration allowlist will be permitted.
3.  **Webhook Validation:** All inbound integration webhooks must be cryptographically validated. For example:
    *   Shopify requests must pass HMAC validation checks using the shared webhook client secret.
    *   Zoho CRM payloads must pass signature checks.
    *   Validated webhooks are instantly converted to internal events and pushed to NATS JetStream.

## Alternatives Considered
1.  **Direct Backend Dispatch:** Rejected. Outbound network requests dispatched directly from the application container without intermediate firewalls/proxies expose the entire internal cloud network to SSRF.
2.  **IP Blacklist Filters in Application Code:** Rejected. Attempting to parse and block IP ranges dynamically in application code is prone to DNS-rebinding bypasses.

## Consequences
*   **SSRF Elimination:** Complete mitigation of SSRF threats at the network architecture layer.
*   **Standardized Integration Adapters:** Webhook payloads are transformed to consistent NATS JetStream event schemas, separating vendor structures from workflow triggers.
*   **Central Egress Logging:** Every outgoing webhook call is logged at the Squid proxy layer, facilitating security audits.

## Risks & Mitigations
*   **Risk:** The forward proxy is a single point of failure. If the proxy goes down, outbound campaigns and integrations stall.
*   **Mitigation:** Deploy the Squid proxy as a multi-node service behind an AWS Network Load Balancer (NLB) with active health checks to automatically recycle failing proxy instances.
