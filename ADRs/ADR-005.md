# ADR-005: Event Strategy

## Status
Approved

## Context
The Conductor modular monolith and the Node.js WhatsApp adapter must communicate asynchronously to handle incoming webhooks, system tasks, and external integration events (Shopify, Zoho CRM). 

We need a messaging broker that:
1.  Provides at-least-once message delivery guarantees (durable streams).
2.  Supports consumer groups for load balancing and concurrent processing.
3.  Has a minimal resource footprint to run cost-effectively in staging and local sandbox environments.

## Decision
We will **Adopt NATS JetStream** as the core event broker and streaming engine:
*   **Topic Hierarchy:** System events will be categorized into distinct subject hierarchies:
    *   `whatsapp.inbound.*`: Inbound messages and status callbacks.
    *   `integration.inbound.shopify`: Inbound Shopify events.
    *   `integration.inbound.zoho`: Inbound Zoho CRM events.
    *   `workflow.events.*`: Workflow lifecycle telemetry.
*   **Durability:** Enable JetStream persistence (using local file storage) to ensure events are not lost in the event of consumer downtime.
*   **Consumer Isolation:** Implement NATS consumer groups to distribute work evenly across JVM monolithic instances.

## Alternatives Considered
1.  **Apache Kafka:** Rejected. Kafka's JVM footprint, complex deployment configurations (ZooKeeper or KRaft), and high memory demands make it too heavy for local developer sandbox machines and MVP production tiers.
2.  **RabbitMQ:** Rejected. NATS JetStream is lighter (~20MB memory footprint), faster, and offers simpler configuration APIs for clustering and topic-based streams.

## Consequences
*   **Extremely Low Overhead:** NATS JetStream runs efficiently with minimal resource utilization.
*   **High Performance:** Sub-millisecond latency for event distribution, supporting rapid response targets (such as the 5-second STOP keyword SLA).
*   **Simplified Sandbox:** Running NATS locally is as simple as launching a single Docker container using ~10MB RAM.

## Risks & Mitigations
*   **Risk:** Stream disk exhaustion if message consumers go offline and persistent queues accumulate messages.
*   **Mitigation:** Define strict stream retention limits (e.g., maximum stream age of 3 days or maximum stream size of 10GB) and direct discarded/undelivered messages to a Dead Letter Queue (DLQ).
