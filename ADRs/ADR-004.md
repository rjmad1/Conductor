# ADR-004: Messaging Architecture

## Status
Approved

## Context
WhatsApp is the primary communications channel for Conductor campaigns. The platform must handle two distinct messaging flows:
1.  **Inbound Webhooks:** Meta pushes message delivery status updates (`sent`, `delivered`, `read`, `failed`) and user reply text to our endpoints. These webhook payloads can spike heavily during active campaigns.
2.  **Outbound Dispatch:** Custom workflow rules trigger outgoing template messages.

Additionally, to ensure compliance with WhatsApp Business policies and prevent account suspensions, the platform must process "STOP" opt-out requests within a strict 5-second window. Utilizing unofficial WhatsApp Web scraping packages (e.g. Baileys, OpenWA) violates Meta's Terms of Service, posing a critical business survival risk.

## Decision
We will base the messaging channel on the **Official Meta WhatsApp Cloud API**:
1.  **Strict Compliance:** Standardize entirely on the official Meta Graph API. All unofficial web-scraping libraries are banned.
2.  **Decoupled Node.js Adapter:** A dedicated, lightweight Node.js/TypeScript micro-service (WhatsApp Adapter) will expose webhook endpoints to receive Meta's payload.
    *   It will perform HMAC SHA-256 signature verification using the pre-configured Meta app secret.
    *   It will immediately publish the parsed payload as a NATS JetStream event and return an HTTP `200 OK` to Meta within 1 second.
3.  **Outbound API Wrapper:** Outbound messages will be sent directly from the Spring Boot monorepo via a resilient HTTP client wrapping Meta's Graph API endpoints.
4.  **Priority STOP Interceptor:** Inbound replies will be parsed. If the message matches unsubscribe keywords (e.g. "STOP", "UNSUBSCRIBE"), the WhatsApp Adapter publishes a high-priority event that updates the Customer Domain's consent status in PostgreSQL within 5 seconds, short-circuiting standard workflow evaluation.

## Alternatives Considered
1.  **Unofficial WhatsApp APIs (Baileys, OpenWA):** Rejected. High likelihood of Meta blacklisting the sender's phone numbers.
2.  **Intermediate Aggregators (Twilio, Gupshup):** Rejected for MVP. Standard aggregator fees double the cost per message compared to Meta's direct pricing, reducing the margin potential for Conductor.

## Consequences
*   **High Resilience:** The Node.js adapter can process thousands of webhooks per second because it only performs signature checks and writes directly to NATS, bypassing heavy business logic.
*   **Regulatory Safety:** Full compliance with Meta Business Policies ensures account longevity.
*   **Low Operating Cost:** Bypassing Twilio/Gupshup ensures direct Meta pricing, maximizing SaaS margins.

## Risks & Mitigations
*   **Risk:** Spikes in Meta webhooks could saturate NATS or PostgreSQL.
*   **Mitigation:** Configure NATS JetStream stream limits and use consumer rate limit controls. Outbound campaign dispatches should be throttled to align with incoming webhook capacity.
