# ADR-010: Deployment Strategy

## Status
Approved

## Context
Conductor must deploy across multiple environments (Local, Dev, QA, Staging, Production). Staging and Production must be resilient, secure, scale horizontally to absorb message traffic peaks, and reside entirely in the AWS Mumbai region (`ap-south-1`) to comply with DPDP India 2023 data residency requirements.

We need a deployment strategy that minimizes physical server administration (operating system patching, hardware provisioning) to reduce operational overhead for a small engineering team, while avoiding the high costs of complex Kubernetes clusters.

## Decision
We will standardize on **Docker-Compose** for local developer sandboxes and **AWS ECS Fargate** for cloud environments:
1.  **Local Environment:** Developers run a single `docker-compose.yml` file spinning up Keycloak, NATS JetStream, Temporal, PostgreSQL, Redis, and the Node.js WhatsApp adapter, alongside the Spring Boot modular monolith.
2.  **AWS Infrastructure (ECS Fargate):** Custom services (Spring Monolith, Node.js WhatsApp webhook receiver) run as serverless container tasks on AWS ECS Fargate. 
3.  **Third-Party Assemblies:** Keycloak, Metabase, and Temporal Server are deployed as official Docker containers on ECS Fargate.
4.  **Database & Caching:** We use AWS RDS PostgreSQL (Multi-AZ in production, single-instance in staging) and AWS ElastiCache Redis.
5.  **Event Streaming:** NATS JetStream is hosted as a containerized service on ECS Fargate utilizing AWS EFS (Elastic File System) for persistent stream storage.
6.  **Secrets Management:** All database passwords, API credentials, and JWT keys are stored in AWS Parameter Store or AWS Secrets Manager. Secrets are injected at launch time as environment variables to the ECS tasks. Hardcoded secrets are blocked in CI via Gitleaks scans.
7.  **CI/CD Pipeline:** GitHub Actions builds Docker images, runs automated test suites, pushes images to AWS Elastic Container Registry (ECR), and triggers AWS ECS task definitions rolling updates.

## Alternatives Considered
1.  **Kubernetes (Amazon EKS):** Rejected for Phase 1. Kubernetes requires continuous node management, ingress configurations, certificate managers, and dedicated operations support. The EKS control plane cost alone (~$73/month per cluster) plus worker node pricing exceeds our start-up budget.
2.  **PaaS (Heroku / Render):** Rejected. PaaS platforms do not easily support running complex state engines like Temporal and NATS JetStream with persistent file systems, or custom egress proxies like Squid.

## Consequences
*   **Minimal Server Management:** ECS Fargate handles container compute scaling, OS patching, and hardware recycling automatically.
*   **Parity Across Environments:** The local Docker-compose environment matches the container configurations running in staging and production, minimizing "works on my machine" bugs.
*   **Strict Regional Compliance:** All AWS RDS, Fargate, and Redis services are locked to the `ap-south-1` region, meeting DPDP guidelines.

## Risks & Mitigations
*   **Risk:** Cost creep from Fargate tasks and RDS database instances.
*   **Mitigation:** Configure CPU/Memory sizing conservatively for the staging environment (e.g. 0.25 vCPU, 512MB RAM per Fargate task). Use RDS Aurora Serverless v2 for staging database to automatically scale down to zero when inactive.
