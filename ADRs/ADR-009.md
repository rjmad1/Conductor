# ADR-009: Audit Architecture

## Status
Approved

## Context
Compliance regulations (such as **SOC2** and **DPDP India 2023**) mandate that Conductor maintain an immutable, tamper-evident audit ledger of all key system operations. This includes:
*   User permissions modifications.
*   System configuration and API key changes.
*   Workflow definition changes.
*   Tenant creation and deletion.
*   Consent changes.

This audit trail must survive tenant deletion requests to ensure compliance. If auditing is implemented purely at the application layer (e.g., using AOP aspects or Hibernate Interceptors), direct database updates executed by administrators, cron scripts, or migration tasks would bypass the audit trail.

## Decision
We will implement a **Database Trigger-Based Audit Ledger** within PostgreSQL:
1.  **Database Triggers:** Enable row-level `AFTER INSERT OR UPDATE OR DELETE` database triggers on all critical configuration tables (including `users`, `workflows`, `api_keys`, `consent_records`, `tenants`).
2.  **Unified Audit Table:** These triggers will write records to a centralized `audit_logs` table.
    *   Schema contents: `id` (UUID), `tenant_id` (UUID), `user_id` (UUID), `action` (VARCHAR), `table_name` (VARCHAR), `row_id` (VARCHAR), `pre_image` (JSONB - state before update), `post_image` (JSONB - state after update), and `created_at` (TIMESTAMP).
3.  **Physical Partitioning:** The `audit_logs` table will be physically partitioned by date (monthly partitions) to allow efficient pruning and scaling.
4.  **Write-Once Immutability:** Enforce write-once constraints at the database level by binding a trigger on `audit_logs` that rejects any `UPDATE` or `DELETE` statements.

```sql
CREATE OR REPLACE FUNCTION block_audit_log_alteration() 
RETURNS TRIGGER AS $$
BEGIN
    RAISE EXCEPTION 'Audit logs are immutable ledgers and cannot be updated or deleted.';
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER enforce_immutable_audit
BEFORE UPDATE OR DELETE ON audit_logs
FOR EACH ROW EXECUTE FUNCTION block_audit_log_alteration();
```

## Alternatives Considered
1.  **Application-Level Logging (Spring Boot AOP):** Rejected. If a support engineer updates an database record via command-line, or a background script updates user profiles, no audit logs are written, creating security and compliance gaps.
2.  **Event Sourcing (e.g., Kafka Connect to Elasticsearch):** Rejected. Setting up and securing a multi-node Elasticsearch or third-party ledger system introduces excessive infrastructure overhead for the initial phase.

## Consequences
*   **100% Audit Coverage:** Every modification to configuration tables is captured, regardless of whether it originates from the web dashboard, API keys, or manual SQL execution.
*   **Zero Application Boilerplate:** Developers do not need to write log statements or call audit services in their Spring controllers.
*   **Compliance Ready:** High-fidelity records directly support SOC2 audit demands.

## Risks & Mitigations
*   **Risk:** Trigger execution increases write latency for database updates.
*   **Mitigation:** The audit trigger will execute minimal logic, converting record changes directly to JSONB strings. Heavy write tables (like message delivery statuses) will be audited via separate application telemetry to NATS rather than database triggers.
