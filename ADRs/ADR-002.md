# ADR-002: Multi-Tenancy Strategy

## Status
Approved

## Context
Conductor is a SaaS application designed for SMBs and mid-market organizations. The system must isolate each tenant's customers, contacts, credentials, templates, and campaign statistics. 

We must choose a multi-tenancy model that guarantees strict data privacy (preventing cross-tenant leaks) while maintaining low infrastructure costs and straightforward operational maintenance (database migrations and schema changes).

## Decision
We will implement a **Shared Database with Logical Partitioning (Row-Level Isolation)** for the Conductor MVP:
1.  **Logical DB Partitioning:** A single PostgreSQL database instance is shared across all tenants. Every table containing tenant-specific data must include a `tenant_id` column as a foreign key referencing the `tenants` table.
2.  **Automated JPA Filters:** Enforce tenant isolation dynamically at the ORM layer using Hibernate `@FilterDef` and `@Filter`. A Spring interceptor will intercept incoming HTTP requests, extract the validated `X-Tenant-ID` from the request context, and set the Hibernate tenant filter parameter for that execution thread. All ORM-generated SELECT, UPDATE, and DELETE operations will automatically append `AND tenant_id = :tenantId`.
3.  **Keycloak Realms:** User credentials and login sessions will be isolated using dedicated Keycloak Realms per tenant, providing identity-level segregation.

## Alternatives Considered
1.  **Database-per-Tenant:** Rejected. Provisioning separate database instances (or RDS databases) for every SMB customer is cost-prohibitive, resulting in massive underutilization of resources.
2.  **Schema-per-Tenant:** Rejected for the initial phase. Running migrations (e.g., Flyway/Liquibase) across hundreds of database schemas sequentially is slow and prone to mid-run failures. Aggregate queries for platform metrics also become overly complex.

## Consequences
*   **Cost Efficiency:** Maximum density of tenants per database instance, reducing running RDS instances costs.
*   **Simple Migrations:** Database schema changes are executed once against the single database, ensuring instant feature updates for all tenants.
*   **Developer Guardrails:** The automatic ORM filter prevents developers from having to manually write tenant filter clauses in every repository query, reducing the risk of accidental leaks.

## Risks & Mitigations
*   **Risk:** Data leakage via native SQL queries that bypass the JPA/Hibernate filter.
*   **Mitigation:** Enforce static analysis rules (e.g., SonarQube, ArchUnit) to flag and block the use of raw `NativeQuery` execution in Spring Data JPA repositories unless explicitly approved. Implement integration tests that verify that a request with Tenant A's token cannot read seeded data belonging to Tenant B.
