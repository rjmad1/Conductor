# ADR-003: Workflow Runtime

## Status
Approved

## Context
A key capability of Conductor is executing multi-step marketing and engagement workflows (e.g. "Trigger: Shopify order placed -> Wait 10 minutes -> Send WhatsApp order confirmation -> Wait 2 days -> Check if order is delivered -> Send review request"). 

Executing these workflows requires:
1.  Handling execution states across system restarts and crashes.
2.  Scheduling stateful sleep delays ranging from minutes to days.
3.  Managing failures, retries, and compensation logic (e.g., updating statuses on failure).
4.  Supporting runtime visibility and execution logs for troubleshooting.

Building a custom workflow engine from scratch involves managing database-backed queues, cron schedulers, thread pools, and complex lock mechanics. This represents a significant risk and high development cost.

## Decision
We will **Adopt Temporal Server** as the core workflow execution engine. 
*   **Workflow Modeling:** Workflows will be defined in a JSON DSL, which the Conductor platform parses and validates.
*   **Temporal Integration:** The Java Spring Boot application will implement Temporal Workflow definitions and Temporal Activities (e.g. `SendWhatsAppActivity`, `EvaluateConditionActivity`, `RazorpayBillingActivity`).
*   **Execution Runtime:** When a workflow is triggered, Conductor initializes a Temporal Workflow Execution. The Temporal Server coordinates task queue scheduling, timers, retries, and state persistence. The Conductor JVM processes act as worker nodes executing the corresponding activities.

## Alternatives Considered
1.  **Custom DB-Backed Engine:** Rejected. Writing custom scheduler state machines in SQL/PostgreSQL or using Spring Scheduled pools requires extensive thread locking, transaction management, and state logic, creating significant technical debt.
2.  **Quartz Scheduler / Spring Batch:** Rejected. These frameworks lack native support for durable, stateful execution graphs with complex dynamic branches, making tracking long-running execution states difficult.

## Consequences
*   **Developer Focus:** The team writes basic activity functions without worrying about state persistence, threading, timer management, or retries.
*   **Durable Executions:** Temporal guarantees that if a Conductor node crashes mid-workflow, the workflow will resume from the exact same step on another active node.
*   **Built-in Observability:** The Temporal Web UI provides out-of-the-box tracking of active and historical workflow state paths.

## Risks & Mitigations
*   **Risk:** Operations overhead of running Temporal Server clusters.
*   **Mitigation:** For the MVP, run the Temporal Server inside a single AWS ECS Fargate container using a dedicated schema in the shared RDS PostgreSQL instance. This isolates its operational overhead without requiring Cassandra or Elasticsearch.
